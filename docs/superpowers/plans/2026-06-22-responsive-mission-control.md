# Responsive Mission Control — Implementation Plan (Component 4 / Plan B)

> **For agentic workers:** execute task-by-task. Unlike the foundation plan (TDD), responsive CSS is visual — each task's gate is: `make ui` builds clean AND a rendered screenshot at phone (≈390px) and tablet (≈820px) widths shows the intended layout (via Playwright against the dev server). Steps use `- [ ]`.

**Goal:** Make the existing Mission Control PWA usable on phones and tablets at all screen sizes, with a proper touch-scrollable terminal that survives the mobile soft keyboard — so the remote-connect feature (foundation plan, already merged on this branch) is actually operable from a phone.

**Architecture:** Mobile-first additions layered on the existing desktop CSS (`internal/server/ui/src/styles/app.css`, 188KB, 17 max-width media queries, smallest 680px). Add a phone breakpoint (≤640px) and tablet breakpoint (≤900px). Below 640px: hide the rail, show a bottom tab bar + drawer, stack panes, convert dense lists to cards, wrap wide content in scroll containers. The terminal gets `visualViewport`-driven keyboard handling + an accessory key row.

**Tech Stack:** React + TypeScript + Vite, xterm.js, plain CSS (no CSS-in-JS — all styles in `src/styles/*.css`).

## Global Constraints

- **No new external dependencies / no CDN** (self-contained PWA). Pure CSS + existing React.
- **Do not regress desktop.** All new behavior lives behind `@media (max-width: …)` or feature checks; desktop (≥901px) layout must be byte-for-byte unchanged.
- **Commit source only** (per the foundation convention): `internal/server/ui/src/**` + CSS. Do NOT commit `internal/server/static/` (gitignored bundle, regenerated by `make ui`; a final rebuild commits the trackable parts).
- **Touch targets ≥44px**; respect `prefers-reduced-motion` (already honored in base.css).
- **Safe areas:** use `env(safe-area-inset-*)` on the topbar (top) and bottom nav (bottom); the viewport meta already has `viewport-fit=cover`.
- **Breakpoints (introduce as shared values):** `--bp-phone: 640px`, `--bp-tablet: 900px` (define in tokens.css; the existing 680/720/1000/1080 queries stay).
- **Verification:** `cd internal/server/ui && pnpm run build` must pass with no TS errors after every task; then render via the dev server and screenshot at 390px and 820px.

### Task R3: Terminal mobile — soft keyboard, scroll, accessory keys (PRIORITY)

**Files:** `src/components/Terminal.tsx`, `app.css` (terminal mobile rules + accessory row).

- [ ] **Soft-keyboard handling:** use `window.visualViewport` (`resize`/`scroll` events) to compute the visible height; set the terminal container height to the visible viewport minus chrome so the prompt is never hidden behind the keyboard. On viewport change, call the existing fit/refit path (`Terminal.tsx:315–353`) so xterm reflows. Guard for browsers without `visualViewport` (fallback to current behavior).
- [ ] **Touch scroll:** ensure xterm's viewport scrolls with touch (it does natively); make sure no parent has `overflow:hidden`/`touch-action:none` blocking it at phone widths. Keep follow-tail behavior (`Terminal.tsx:170–194`).
- [ ] **Scroll-to-bottom button** (`app.css:1628–1663`): relocate above the accessory row / keyboard at ≤640px so it isn't hidden.
- [ ] **Accessory key row:** a horizontally-scrollable row of buttons shown only at ≤640px (or when the terminal has focus on touch), each sending the right control sequence to the PTY via the existing input path (the `term.onData`/send mechanism): `Esc` (`\x1b`), `Tab` (`\t`), `Ctrl` (sticky modifier — next key sent as control char), `↑↓←→` (`\x1b[A`/`B`/`C`/`D`), `/`, `|`, and a paste affordance. 44px targets. Sits directly above the soft keyboard.
- [ ] **Default cols/rows:** at phone widths, don't force 120 cols — let the FitAddon size to the actual narrow viewport (it already fits; just don't hardcode a wide default on small screens).
- [ ] **Verify:** build; screenshot the terminal at 390px (full-bleed, accessory row visible); if feasible, simulate focus to confirm the accessory row renders. Note: real soft-keyboard behavior needs a physical device — document that for the operator's manual check.
- [ ] Commit source.

## Notes on verification tooling

Visual verification uses the Vite dev server (`pnpm dev` on :5173, proxies API/WS to a running `flow ui serve` on :8787) driven by Playwright (`browser_navigate` + `browser_resize` to 390×844 / 820×1180 + `browser_take_screenshot`). The dev server renders source directly (HMR) — no binary rebuild needed per iteration. If the backend/token isn't available, the shell/nav/terminal *chrome* and empty states still render for layout verification; data-dependent screens verify against whatever the running backend has.
