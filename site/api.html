API reference

Mission Control (flow ui serve) exposes a local HTTP + WebSocket API — the same surface the browser SPA uses. It's a local-first, loopback-only API: ~86 HTTP endpoints and a handful of WebSocket channels, all gated by a same-origin check and a per-boot session token.

No OpenAPI spec — yet. The API is hand-implemented in Go (routes registered in internal/server/server.go). This page is the authoritative reference, maintained by hand. Shapes can change between alpha releases; pin a version if you script against it.

About the API

The server is a single Go binary with no Node runtime — the static UI is embedded at compile time and served from /. Every read and mutation the browser performs is available here; in fact the SPA runs all of them over a single WebSocket-RPC connection rather than plain fetch.

Base URL

http://127.0.0.1:8787

Default host/port. Override with flow ui serve --host <h> --port <p>. The server refuses to bind a non-loopback host unless you pass --host explicitly — there is no built-in auth or TLS, so put your own proxy in front if you expose it.

Authentication

Two layers guard the API. Both are enforced in internal/server/session_token.go.

1 · Same-origin gate

WebSocket upgrades require an Origin header that exactly matches the server host. Empty origins (non-browser clients) are rejected on /ws/*.

2 · Session token

At boot the server mints a random token (crypto/rand) and writes it to ~/.flow/.ui-session-token (mode 0600). It guards every state-changing /api/* request and every WebSocket handshake.

  • HTTP — send it as the X-Flow-Session-Token header.
  • WebSocket — pass it as the token query parameter (browsers can't set custom headers on a WS handshake).
  • The browser SPA reads it from window.__FLOW_TOKEN__, injected into index.html. Trusted CLIs read the token file directly.
$ TOKEN=$(cat ~/.flow/.ui-session-token)
$ curl -s http://127.0.0.1:8787/api/tasks \
    -H "X-Flow-Session-Token: $TOKEN"

Token-exempt routes (authenticated by other means, still origin-checked): /api/github/webhook (HMAC), /api/hooks/agent & /api/inbox/notify (loopback-only), and the OAuth callbacks /api/slack/oauth/callback & /api/github/setup/callback (state-nonce validated). Read-only GETs are reachable same-origin; mutations always require the token.

REST

Standard JSON over HTTP. GET reads; POST/PUT/PATCH/DELETE mutate and require the token. Markdown briefs and updates are sent/returned as text/markdown. Path params are written {slug} below.

WebSocket RPC — WS /ws/rpc

The primary transport. Run any /api/* request over one bidirectional socket. Auth: origin match + ?token=. On connect the server sends {"type":"ready", "server_version":"…"}; it pings every 25s.

Client → server

{
  "id": "req-123",
  "method": "POST",
  "path": "/api/actions",
  "body": { "kind": "spawn", "target": "my-task" },
  "text": "raw markdown",          // for markdown PUT (optional)
  "content_type": "text/markdown",
  "files": [{ "field": "files", "name": "x.png",
              "content_type": "image/png", "data": "<base64>" }]
}

Server → client

{
  "type": "rpc",
  "id": "req-123",
  "status": 200,
  "content_type": "application/json",
  "json": { },                     // parsed JSON response
  "text": "…",                     // non-JSON body
  "error": "…"                     // on failure
}

File uploads are base64-encoded in the frame; the bridge rebuilds real multipart/form-data so handlers work unchanged (max 96 MiB per frame).

Event stream — WS /ws/events

Real-time fan-out of agent hooks, UI-change signals, liveness snapshots, and steering stage progress. Optional query filters: session_id, task_slug, types=agent_hook,ui_change,…. There is also an SSE-over-HTTP variant at GET /api/events and GET /api/work-events.

{ "type": "subscribed", "subscriber_id": "sub-1", "hook_version": 42 }
{ "type": "agent_hook", "session_id": "…", "task_slug": "…",
  "hook": { "provider": "claude-code", "kind": "PostToolUse" } }
{ "type": "ui_change", "data": { "kind": "tasks" } }
{ "type": "liveness", "task_slug": "…",
  "liveness": { "status": "alive", "reason": "…" } }
{ "type": "ping", "ts": "2026-06-15T10:30:30Z" }

Backpressure: a 64-event buffer per subscriber; slow clients are dropped rather than blocking the publisher.

Terminal bridge — WS /ws/terminal

PTY I/O between the browser terminal widget and the task's tmux session. Query: ?slug=<task>&cols=120&rows=32. The floating (Ask Flow) variant is /ws/floating-terminal?id=<id>; both share the frame format.

// client → server
{ "type": "input", "data": "ls -la\n" }
{ "type": "resize", "cols": 120, "rows": 32 }

// server → client
{ "type": "output", "data": "…" }
{ "type": "state", "data": "live|exited", "message": "…" }

Endpoints · Actions

Most mutations funnel through one verb-dispatched endpoint:

MethodPathPurpose
POST/api/actionsExecute an action by kind

The request body's kind selects the operation — create-flow, spawn, resume, fork, update-settings, and more. Common fields:

{
  "kind": "spawn",            // create-flow | spawn | resume | fork | update-settings …
  "target": "<slug>",         // task/project to act on
  "name": "Add OAuth login",
  "project": "<slug>",
  "work_dir": "/path/to/repo",
  "priority": "high",
  "prompt": "first message for the session",
  "provider": "claude-code", // or codex
  "model": "opus",
  "attention_action": "make-task",
  "settings": { }            // for update-settings
}

Response: { "ok": true, "message": "…", "agent": {…}, "bridge": true }. The full field set lives in internal/server/actions.go and types.go.
