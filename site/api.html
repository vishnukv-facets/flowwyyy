API reference

Mission Control (flow ui serve) exposes a local HTTP + WebSocket API — the same surface the browser SPA drives. It's local-first and loopback-only: ~86 HTTP endpoints and four WebSocket channels, gated by a same-origin check and a per-boot session token. Every shape below is taken from the Go structs in internal/server/.

No OpenAPI spec — yet. The API is hand-implemented in Go (routes in internal/server/server.go). This page is the authoritative reference. Shapes may change between alpha releases; pin a version if you script against it.

About the API

The server is a single Go binary — no Node runtime — that embeds the static UI at compile time and serves it from /. Every read and mutation the browser performs is available here; the SPA itself runs them over a single WebSocket-RPC connection rather than plain fetch.

Base URL

http://127.0.0.1:8787

Default host/port. Override with flow ui serve --host <h> --port <p>. The server refuses a non-loopback host unless you pass --host explicitly — there's no built-in auth or TLS, so front it with your own proxy if you expose it.

Authentication

Two layers, both in internal/server/session_token.go.

1 · Same-origin gate

WebSocket upgrades require an Origin header that exactly matches the server host. Empty origins (non-browser clients) are rejected on /ws/*.

2 · Session token

At boot the server mints a random token (crypto/rand) at ~/.flow/.ui-session-token (mode 0600). It guards every state-changing /api/* request and every WS handshake.

  • HTTP — send it as the X-Flow-Session-Token header.
  • WebSocket — pass it as the token query parameter (browsers can't set custom WS headers).
  • The SPA reads it from window.__FLOW_TOKEN__; trusted CLIs read the token file.
$ TOKEN=$(cat ~/.flow/.ui-session-token)
$ curl -s http://127.0.0.1:8787/api/tasks \
    -H "X-Flow-Session-Token: $TOKEN"

Token-exempt (authenticated by other means, still origin-checked): /api/github/webhook (HMAC), /api/hooks/agent & /api/inbox/notify (loopback-only), and the OAuth callbacks /api/slack/oauth/callback & /api/github/setup/callback (state-nonce validated).

Conventions

  • Content type — JSON in and out, except briefs/updates which are text/markdown on GET/PUT.
  • Optional fields — fields tagged omitempty are dropped from the response when empty; treat any field below as possibly absent unless it's a core identifier.
  • Errors — a non-2xx status with body { "error": "message" }. See Error.
  • Enums — status backlog | in-progress | done; priority high | medium | low; provider claude-code | codex.

REST

Standard JSON over HTTP. GET reads; POST/PUT/PATCH/DELETE mutate and require the token. Path params are written {slug}.

WebSocket RPC — WS /ws/rpc

The primary transport: run any /api/* request over one bidirectional socket. Auth: origin match + ?token=. On connect the server sends a ready frame; it pings every 25s.

Greeting (server → client)
{ "type": "ready", "server_version": "string", "flow_root": "string" }
Request frame (client → server)
{
  "id": "req-123",
  "method": "GET | POST | PUT | DELETE",
  "path": "/api/actions",
  "body": { },                  // JSON body (optional)
  "text": "raw markdown",       // for text/markdown PUT (optional)
  "content_type": "text/markdown",
  "form": { "key": "value" },   // form fields (optional)
  "files": [ { "field": "files", "name": "x.png",
              "content_type": "image/png", "data": "<base64>" } ]
}
Response frame (server → client)
{
  "type": "rpc",
  "id": "req-123",
  "status": 200,
  "content_type": "application/json",
  "json": { },     // parsed JSON response
  "text": "…",    // non-JSON body
  "error": "…"    // on failure
}

File uploads are base64-encoded in the frame; the bridge rebuilds real multipart/form-data so handlers work unchanged (max 96 MiB per frame).

Event stream — WS /ws/events

Real-time fan-out of agent hooks, UI-change signals, liveness snapshots, and steering progress. Filters: ?session_id=, ?task_slug=, ?types=agent_hook,ui_change,…. SSE-over-HTTP variants live at GET /api/events and GET /api/work-events.

{ "type": "subscribed", "subscriber_id": "sub-1", "hook_version": 42 }
{ "type": "agent_hook", "task_slug": "…",
  "hook": { "provider": "claude-code", "kind": "PostToolUse" } }
{ "type": "ui_change", "data": { "kind": "tasks" } }
{ "type": "liveness", "task_slug": "…",
  "liveness": { "status": "alive", "reason": "…" } }
{ "type": "ping", "ts": "2026-06-15T10:30:30Z" }

Backpressure: a 64-event buffer per subscriber; slow clients are dropped rather than blocking the publisher.

Terminal bridge — WS /ws/terminal

PTY I/O between the browser terminal and the task's tmux session. Query ?slug=<task>&cols=120&rows=32. The floating (Ask Flow) variant is /ws/floating-terminal?id=<id>; same frames.

// client → server
{ "type": "input", "data": "ls -la\n" }
{ "type": "resize", "cols": 120, "rows": 32 }

// server → client
{ "type": "output", "data": "…" }
{ "type": "state", "data": "live | exited", "message": "…" }

Actions

Most mutations funnel through one verb-dispatched endpoint. The kind field selects the operation; only the fields relevant to that kind are read.

Request
{
  "kind": "spawn",           // see kinds below
  "target": "<slug>",        // task/project to act on
  "slug": "<new-slug>",      // for create-* kinds
  "name": "Add OAuth login",
  "description": "…",
  "project": "<slug>",
  "work_dir": "/path/to/repo",
  "path": "…",
  "priority": "high | medium | low",
  "prompt": "first message for the session",
  "provider": "claude-code | codex",
  "model": "opus | sonnet | haiku | …",
  "permission_mode": "default | auto | bypass",
  "session_id": "<uuid>",
  "branch": "…",
  "mkdir": false,
  "no_open": false,
  "schedule": "every day at 9am",
  "schedule_op": "set | clear | pause | resume",
  "attention_action": "make_task | forward | dismiss | …",
  "reply_text": "…",
  "reply_instructions": "…",

Response · 200

Accepted kind values

Sessions: spawn, resume, attach, restart, restart-fresh, kill, nudge, fork, switch-branch, iterm/terminal/warp/kitty/ghostty/wezterm/tmux/vscode/alacritty · Lifecycle: archive, unarchive, done, delete, restore, destroy, empty-trash, pause, clear-waiting, mark-read, approve, deny · Create: create-flow, create-project, create-playbook, create-kb, spawn-run · Update: update-priority, update-provider, update-model, update-permission-mode, update-task-name, update-project, update-playbook, edit-playbook, set-playbook-schedule · Workdirs: workdir-add, workdir-rename, workdir-remove · Attention: attention-act · System: update-settings, rotate-webhook-secret, rotate-ingress-url, reveal-webhook-secret, compact-db · Chats: chat-archive, chat-unarchive, chat-delete, chat-reopen, overview-chat, close-floating-terminal.

Connectors & system

Setup/status endpoints return small status objects; mutations (create-app, OAuth, disconnect) run through the wizards. The full set:

MethodPathPurpose
GET/api/slack/setup/statusSlack connection + wizard state
POST/api/slack/setup/create-appCreate the Slack app from a manifest; /oauth/start, /app-token, /oauth/cancel, /reset round it out
GET/api/slack/channelsChannels the bot is in
POST/api/slack/sendSend to a channel/thread (gated by write perms)
GET/api/github/setup/statusApp install status + wizard state
POST/api/github/setup/create-appBuild + POST the GitHub App manifest; /installations, /orgs, /backfill, /disconnect manage it
POST/api/github/webhookWebhook ingestion — HMAC-signed (X-Hub-Signature-256), token-exempt
GET/api/github/webhook/statusDelivery + signature status
GET/api/ingress/statusPublic ingress URL + provider
GET/api/kbList KB files; /{file} reads/PUTs one; /kb/dream runs a hygiene pass
GET/api/brain/graphKnowledge graph (nodes, edges); /node/{id} for detail
GET/api/workdirs · /api/tagsKnown repos; tags + usage counts
POST/api/fs/entries · /api/fs/mkdirList a directory; create one
