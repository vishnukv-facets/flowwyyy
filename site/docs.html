Documentation

flowwyyy is the flow CLI plus a browser-based Mission Control. Install once, say "let's get to work," and the bundled skill turns plain English into flow commands. This page covers install, every command and its arguments, connector setup, and where your data lives. References (<ref>) resolve by exact slug.

Install

flowwyyy runs on macOS + Claude Code or Codex. The CLI binary is named flow.

Homebrew (recommended)

$ brew install vishnukv-facets/flowwyyy/flowwyyy
$ flow init

Taps vishnukv-facets/homebrew-flowwyyy and installs flow. Upgrade later with brew upgrade flowwyyy. The formula is named flowwyyy to avoid colliding with homebrew-core's flow; the command it installs is still flow.

Manual (curl + tar)

# Apple Silicon — use amd64 for Intel
$ ARCH=arm64
$ curl -fsSL \
    "https://github.com/vishnukv-facets/flowwyyy/releases/latest/download/flow-darwin-${ARCH}.tar.gz" \
    | tar -xz -C /usr/local/bin flow
$ chmod +x /usr/local/bin/flow
$ xattr -d com.apple.quarantine /usr/local/bin/flow 2>/dev/null || true
$ flow init

The xattr step removes Gatekeeper's quarantine attribute so macOS doesn't refuse to run the unsigned binary.

What flow init does

  • Creates ~/.flow/ — the SQLite index, knowledge base, and project/task/playbook/owner directories.
  • Writes the flow skill to ~/.claude/skills/flow/SKILL.md.
  • Adds a SessionStart hook to ~/.claude/settings.json so every new session auto-loads the skill.
Upgrading? In any Claude Code session say "Upgrade flowwyyy from https://github.com/vishnukv-facets/flowwyyy" — it fetches the latest binary and runs flow skill update. Check your version with flow --version.

First run

Open Claude (or Codex) and say "let's get to work." The skill handles the rest — capturing work as briefs, resuming sessions, and tracking what you're waiting on. Prefer the browser? Run flow ui serve and open http://127.0.0.1:8787.

The session model

flow do <task> resolves the task's provider, pre-allocates or captures a session id, writes it to the task row, and spawns a tab — running claude --session-id <uuid> (or codex resume). Future flow do calls resume the same conversation, and a SessionStart hook re-injects the brief, updates, and repo context on every resume.

By default the session launches inside a per-task git worktree at <repo>/.<agent>/worktrees/<slug> on branch flow/<slug>, so parallel tasks never step on each other. flow do --here binds the current session instead and never relocates.

Which model a session launches with

You don't pick a model during intake — it's resolved at launch. An explicit --model always wins. Otherwise flow picks a tier (FLOW_MODEL_TIER, default medium), upshifted for high-priority work and downshifted for richly-specified briefs.

TierClaudeCodex
largeopusgpt-5.5
medium (default)sonnetgpt-5.4
smallhaikugpt-5.4-mini

The model is locked once a session starts — change it only while the task is in backlog.

Setup

Create ~/.flow/, initialise the database + knowledge base, and install the skill and SessionStart hook. Required once before anything else.

$ flow init

Manage the bundled skill file on disk.

$ flow skill install [--force]
$ flow skill uninstall
$ flow skill update          # reinstall after upgrading the binary
install --force
Overwrite an existing skill file.
update
Refresh the skill and re-wire the SessionStart / UserPromptSubmit hooks. Use this after upgrading flow.

Create

Create a task. The interview-driven skill normally does this for you, but the flags are:

$ flow add task "<name>" [flags]
--slug <s>
Short, memorable, ASCII slug. Auto-derived from the name if omitted.
--project <slug>
Attach to an existing project. Omit for an adhoc/floating task.
--work-dir <path>
Where the task lives on disk. Defaults to the project's work_dir, or an auto-created workspace.
--mkdir
Create --work-dir if it doesn't exist.
--priority h|m|l
Priority. Default medium.
--due <date>
Due date — YYYY-MM-DD, today, tomorrow, a weekday, or Nd.
--agent claude|codex
Session agent (also --claude / --codex). Locked once a session starts.
--tag <t>
Attach a tag. Repeatable.
--permission-mode default|auto|bypass
Tool-approval mode for the session.

Create a project — an ongoing container that groups related tasks.

$ flow add project "<name>" --work-dir <path> [flags]
--work-dir <path>
Required. The repo/path the project's work happens in.
--slug <s>
Custom slug.
--priority h|m|l
Project priority.
--mkdir
Create the work-dir if missing.

Create a playbook — a reusable run definition (see Playbooks).

$ flow add playbook "<name>" --work-dir <path> [--slug <s>] [--project <slug>] [--mkdir]

Create an owner — a durable, repo-scoped controller that wakes on a cadence (see Owners).

$ flow add owner "<name>" --work-dir <path> [flags]
--work-dir <path>
Required. The repo the owner operates in.
--project <slug>
Attach to a project.
--every <dur>
Tick cadence, e.g. 24h, 6h.
--agent claude|codex
Agent used for dispatched work.

Sessions

Start or resume work on a task. Spawns a terminal tab inside the task's worktree and flips it to in-progress.

$ flow do <ref> [flags]
--agent claude|codex
Override the stored provider for this launch.
--fresh
Start a brand-new session instead of resuming the stored one.
--dangerously-skip-permissions
Pass the agent's low-friction permissions flag (Codex: bypass approvals + sandbox).
--force
Override the live-session guard (the task already has a running session elsewhere).

Bind this Claude/Codex session to the task — no new tab. Status flips to in-progress; future flow do resumes here.

$ flow do --here <ref> [--force]
--force
Overwrite the target task's prior binding. Cannot rebind a session already bound to another task.

Run a task end-to-end headlessly — no tab, no human. Returns immediately with the supervisor PID + log path; the run closes itself with flow done.

$ flow do --auto <ref> [--with "<instruction>" | --with-file <path>]
--with "<text>"
Inject a one-off instruction as the first message.
--with-file <path>
Point the run at an instruction file instead.

No human watches an --auto run — match the model to the task with --model (set at creation), or set --priority high so the resolver upshifts. Block on completion with flow wait.

Close out a task: run the KB close-out sweep, push the worktree branch, and open a PR with the brief as its body.

$ flow done <ref>

Hook handlers — wired into ~/.claude/settings.json by flow init. You rarely call these by hand.

$ flow hook session-start
$ flow hook agent-event --provider claude|codex

Edit & mutate

Open an entity's brief.md in $EDITOR.

$ flow edit <ref>

$ flow update task <ref> [flags]
--slug <new> · --name <new>
Rename the slug or display name.
--work-dir <path> [--mkdir]
Relocate the task on disk.
--status backlog|in-progress|done
Change status (e.g. reopen a done task).
--priority h|m|l
Change priority.
--assignee <name> · --clear-assignee

--due-date <date> · --clear-due
Set / clear the due date.
--parent <task> · --clear-parent
Set / clear a parent-child dependency.
--waiting "<x>" · --clear-waiting

--tag <t> · --remove-tag <t> · --clear-tags

Edit project / playbook fields.

$ flow update project  <ref> [--slug <new>] [--name <new>] [--work-dir <path>] [--mkdir] [--priority h|m|l]
$ flow update playbook <ref> [--slug <new>] [--name <new>] [--work-dir <path>] [--mkdir]

Lifecycle: archive (set-aside history), soft-delete (recoverable), and their inverses. All accept task / project / playbook refs.

$ flow archive   <ref>
$ flow unarchive <ref>
$ flow delete    <ref>          # soft-delete; hides from lists/UI
$ flow restore   <ref>          # undo a soft-delete

Attention

The Attention Router is the cross-source triage queue. Review it as an operator — it surfaces, it doesn't autonomously act. (Full feed UI lives in Mission Control; see the API.)

$ flow attention list [--status new|acted|dismissed|all]
$ flow attention act <id> <make-task|forward|dismiss>
$ flow attention feedback [--group <dimension>]
list --status
Filter cards by state. Default new.
act <id> <action>
make-task (tracked work), forward (route to a matched task), or dismiss (hide; doesn't mute the source).
feedback --group
Approval/dismiss rates by source|channel|author|thread-type|suggested-action|confidence-band.

Owners

Owners orchestrate ongoing outcomes via short ticks — they dispatch tasks/runs, never do substantive work inline.

$ flow owner start|pause|retire <slug>
$ flow owner tick <slug> [--auto]
$ flow owner next <slug> (--in <duration> | --at <RFC3339>)
$ flow owner tick-due              # scheduler entry point (host cron)
start|pause|retire
Lifecycle controls.
tick [--auto]
Run a tick now — guided by default, headless with --auto.
next --in <dur> | --at <ts>
Set the next wake (relative duration or absolute RFC3339).

Playbooks

A playbook is a reusable run definition. Each run snapshots the brief into a fresh task with its own session — reproducible, frozen at run time.

$ flow run playbook <slug> [flags]
--agent claude|codex
Provider for this run.
--dangerously-skip-permissions
Low-friction permissions for the spawned session.
--auto
Run headless (no tab), self-closing — the mode scheduled runs use.
--with "<text>" | --with-file <path>
Inject a one-off instruction (only with --auto).
Scheduling. A playbook can fire unattended on a cadence — set it with flow update playbook <slug> --schedule "every day at 9am" (or "every 6 hours", a weekday + time, or raw cron). --pause-schedule, --resume-schedule, and --clear-schedule control it. Scheduled runs always fire in --auto mode.

Orchestration

A small vocabulary for delegating to child agents in other tabs and coordinating between them.

Create a task (optionally parented) and immediately open it in a new tab. The prompt becomes the child's brief "What".

$ flow spawn "<name>" [flags]
--parent <slug>
Parent task; this spawn becomes its (non-blocking) subtask.
--depends-on <slug>
A task this spawn is blocked by. Repeatable.
--prompt "<text>"
Seeds the child's brief What section.
--project <slug> · --work-dir <path> · --slug <s> · --mkdir
Placement, as for add task.
--priority h|m|l · --agent claude|codex · --model <m>
Priority, provider, model override.
--no-open
Create the task but don't spawn a session yet.

Append a message to a task's inbox; a live flow terminal is woken through the inbox monitor.

$ flow tell <slug> "<message>" [flags]
--from <label>
Sender label. Defaults to the caller's bound task slug, else user.
--no-notify
Skip the UI inbox-changed event.

Block until a task reaches a state. Event-driven (subscribes to /ws/events), not polling.

$ flow wait <slug> [flags]
--until <state>
backlog|in-progress|done (lifecycle) or running|waiting|idle|dead|released (runtime). Default done.
--timeout <dur>
Max wait; 0 = forever. Default 5m.
--server <url>
flow UI base URL. Default $FLOW_UI_URL or http://127.0.0.1:8787.

Workdirs

A registry of known local repo paths so the skill can match repo intent to a path on disk.

$ flow workdir list
$ flow workdir add <path> [--name <nickname>]
$ flow workdir remove <path>
$ flow workdir scan [<root>] [--add]
add --name
Register a path with a nickname (captures the git origin remote if present).
scan [--add]
Detect repos under a root; --add backfills origin remotes / registers them.

Slack

Post a message as the bot. Requires FLOW_SLACK_WRITES_ENABLED=1.

$ flow slack send --channel <id> --text "<message>"

Slack — react to triage

flowwyyy listens to a Slack workspace over Socket Mode and turns your reactions into tasks. React to a thread with :claude: for a Claude session, :codex: for Codex.

Quick setup — the Connect Slack wizard. Open Mission Control → Connectors → Slack. Three resumable steps:

  • Create the app from an app-configuration token — flow calls Slack's manifest API and wires every scope, event subscription, and Socket Mode for you.
  • App-level token — a deep-linked paste (Slack has no API for this one).
  • Install — one click round-trips OAuth and hands flow the bot token, your user token (DM following), and your member ID.

Reactions are explicit consent from your account, so a coworker's :claude: is harmless noise — only the IDs in FLOW_SLACK_SELF_USER_IDS count. flow tags each task slack-thread:<channel>:<ts>: one thread, one task, forever.

GitHub — assigned work & review threads

flowwyyy ingests GitHub activity — assigned issues, assigned/review-requested PRs, review comments, head updates, merges, and closes — via signed webhook deliveries from a GitHub App. GitHub POSTs to /api/github/webhook; flow verifies the HMAC, dedupes, normalizes, and dispatches — no GitHub API call on the hot path.

Setup — the Connect GitHub wizard. Mission Control → Connectors → GitHub builds a GitHub App manifest (webhook URL, generated signing secret, the right events + permissions), POSTs it to github.com, and receives the app id, private key, and webhook secret in one shot — secrets go to the OS keyring, nothing pasted by hand. It then guides installation. The wizard requires a running public ingress first.

PR tasks are tagged gh-pr:<owner>/<repo>#<n>, issue tasks gh-issue:…. A CHANGES_REQUESTED review or new head SHA reopens a done task; a merge marks it done. If flow was down, the wizard's Replay missed deliveries button replays the gap, deduped by delivery GUID.

Public ingress

Some connectors need an external HTTPS URL to reach flow inbound — Slack's OAuth redirect and GitHub's webhook POSTs. flow keeps running locally, exposes only the connector callback paths (never the UI or data API), and validates everything itself.

ProviderPublic URL fromUse when
none—Slack-only, accepting the localhost cert warning
zrokgenerated at runtimeyou want a public URL without running a server
manualFLOW_PUBLIC_BASE_URLyou already front flow with your own proxy/tunnel

zrok is recommended — flow embeds the zrok Go SDK (no binary, no subprocess). Enable once with zrok enable <token>, set FLOW_INGRESS_PROVIDER=zrok and a stable FLOW_ZROK_SHARE_NAME, and check live state at GET /api/ingress/status.

Your data — local, portable, yours

Everything lives under ~/.flow/ (override with $FLOW_ROOT). No server, no cloud, no telemetry — plain markdown beside a SQLite index, readable in any editor and versionable in git.

~/.flow/
  flow.db                          # SQLite index — projects, tasks, playbooks, owners
  kb/  user.md org.md products.md processes.md business.md
  projects/<slug>/  brief.md  updates/YYYY-MM-DD-*.md
  tasks/<slug>/     brief.md  updates/*.md  inbox.jsonl
  playbooks/<slug>/ brief.md  updates/*.md
  owners/<slug>/    charter.md updates/*.md

The database is an index, not the source of truth — every entity's real content is the markdown beside it; you could delete flow.db and rebuild from the markdown. Back up with git, Time Machine, or a synced folder (add kb/ to .gitignore if you push to a shared remote). To move machines: copy ~/.flow/ over, install the binary, run flow init once.

⚠️ Don't run flow on two machines simultaneously through a synced folder — SQLite doesn't tolerate concurrent writes from separate hosts.
