Documentation

flowwyyy is the flow CLI plus a browser-based Mission Control. Install once, say "let's get to work," and the bundled skill turns plain English into flow commands. This page covers install, the commands you'll reach for daily, connector setup, and where your data lives.

Install

flowwyyy runs on macOS + Claude Code or Codex. The CLI binary is named flow.

Homebrew (recommended)

$ brew install vishnukv-facets/flowwyyy/flowwyyy
$ flow init

Taps vishnukv-facets/homebrew-flowwyyy and installs flow. Upgrade later with brew upgrade flowwyyy. The formula is named flowwyyy to avoid colliding with homebrew-core's flow; the command it installs is still flow.

Manual (curl + tar)

# Apple Silicon — use amd64 for Intel
$ ARCH=arm64
$ curl -fsSL \
    "https://github.com/vishnukv-facets/flowwyyy/releases/latest/download/flow-darwin-${ARCH}.tar.gz" \
    | tar -xz -C /usr/local/bin flow
$ chmod +x /usr/local/bin/flow
$ xattr -d com.apple.quarantine /usr/local/bin/flow 2>/dev/null || true
$ flow init

The xattr step removes Gatekeeper's quarantine attribute so macOS doesn't refuse to run the unsigned binary.

What flow init does

It's the step that wires flowwyyy into your agent. It:

  • Creates ~/.flow/ — the SQLite index, knowledge base, and project/task/playbook/owner directories.
  • Writes the flow skill to ~/.claude/skills/flow/SKILL.md.
  • Adds a SessionStart hook to ~/.claude/settings.json so every new session auto-loads the skill.
Upgrading? In any Claude Code session say "Upgrade flowwyyy from https://github.com/vishnukv-facets/flowwyyy" — it fetches the latest binary and runs flow skill update. Check your version with flow --version.

First run

Open Claude (or Codex) and say "let's get to work." The skill handles the rest — capturing work as briefs, resuming sessions, and tracking what you're waiting on. Prefer the browser? Run flow ui serve and open http://127.0.0.1:8787.

The session model

flow do <task> resolves the task's provider, pre-allocates or captures a session id, writes it to the task row, and spawns a tab — running claude --session-id <uuid> (or codex resume). Future flow do calls resume the same conversation, and a SessionStart hook re-injects the brief, updates, and repo context on every resume.

By default the session launches inside a per-task git worktree at <repo>/.<agent>/worktrees/<slug> on branch flow/<slug>, so parallel tasks never step on each other. flow do --here binds the current session instead and never relocates.

flow do

Start or resume work on a task. The session opens in a new terminal tab and the task flips to in-progress.

$ flow do <task>                          # start / resume in a new tab
$ flow do <task> --here                   # bind THIS session, no new tab
$ flow do <task> --auto                   # headless run — no tab, no human
$ flow do <task> --with "<instruction>"   # inject a first message
$ flow done <task>                        # close out: KB sweep, push branch, open PR

--auto returns immediately (printing the supervisor PID + log path); Claude runs via claude -p, Codex via codex exec. Because no human is watching, match the model to the task — pin --model or set --priority high. flow wait <task> --until done blocks on completion.

flow list

See what's on your plate. Lists default to active (non-archived, non-deleted) rows.

$ flow list tasks                          # all active tasks
$ flow list tasks --status in-progress     # filter by status
$ flow list tasks --priority high --project <slug>
$ flow list tasks --tag <tag> --since monday
$ flow list projects                       # projects
$ flow list runs --since 7d                 # playbook runs

Status is one of backlog, in-progress, done. Add --include-archived or --deleted to widen. Tasks waiting on someone show a [waiting: …] note; stale ones carry a ⚠ marker.

flow standup

A copyable daily briefing — it separates needs action from FYI and groups by project, source, and urgency.

$ flow standup --for today
$ flow standup --for monday --clipboard    # also copy to clipboard

It assembles Attention cards, waiting work, stale sessions, ready high-priority backlog, and recent activity into one digest you can paste anywhere.

flow owner

Owners are durable, repo-scoped controllers for ongoing outcomes. An owner isn't one long session — it's a charter, a journal, and short ticks that wake, review state, dispatch work, self-pace, and exit.

$ flow add owner "<name>" --work-dir <path> [--every 24h] [--agent claude|codex]
$ flow owner list
$ flow owner show <slug>
$ flow owner start|pause|retire <slug>
$ flow owner tick <slug>                    # guided tick; --auto for headless
$ flow owner next <slug> --in 24h           # set the next wake

Owners orchestrate; they don't do substantive work inline. A tick dispatches tasks or playbook runs that self-close, writes a journal note, sets its next wake, and exits. The scheduler is an in-process heartbeat inside flow ui serve.

flow ui serve

Boots Mission Control — a local web app at 127.0.0.1:8787. Same SQLite, same markdown briefs, same skill; just the richer surface.

$ flow ui serve                            # foreground on :8787
$ flow ui serve --bg                        # detached, logs to ~/.flow/logs/
$ flow ui serve --port 9000 --host 0.0.0.0  # bind elsewhere (see below)

You get side-by-side task lists, an Attention-aware briefing, inline brief editing, live agent status, a Cmd+K palette, and a browser-attached terminal that streams the Claude or Codex session over WebSocket.

Local by design. No auth, no TLS, loopback only — the binary refuses to bind a non-loopback host without an explicit --host flag. If you want it on a network, put your own auth in front. See the API auth model.

Slack — react to triage

flowwyyy listens to a Slack workspace over Socket Mode and turns your reactions into tasks. React to a thread with :claude: and a Claude session spins up bound to that thread; react with :codex: and you get a Codex session.

Quick setup — the Connect Slack wizard. Open Mission Control → Connectors → Slack. The wizard does the whole dance in three resumable steps:

  • Create the app from an app-configuration token — flow calls Slack's manifest API and wires every scope, event subscription, and Socket Mode for you.
  • App-level token — a deep-linked paste (Slack has no API for this one).
  • Install — one click round-trips OAuth and hands flow the bot token, your user token (DM following), and your member ID.

Reactions are explicit consent from your account, so a coworker's :claude: is harmless noise — only the IDs in FLOW_SLACK_SELF_USER_IDS count. flow tags each task slack-thread:<channel>:<ts>: one thread, one task, forever.

GitHub — assigned work & review threads

flowwyyy ingests GitHub activity — assigned issues, assigned/review-requested PRs, review comments, head updates, merges, and closes — via signed webhook deliveries from a GitHub App. GitHub POSTs to /api/github/webhook; flow verifies the HMAC, dedupes, normalizes, and dispatches — no GitHub API call on the hot path.

Setup — the Connect GitHub wizard. Mission Control → Connectors → GitHub builds a GitHub App manifest (webhook URL, generated signing secret, the right events + permissions), POSTs it to github.com, and receives the app id, private key, and webhook secret in one shot — secrets go to the OS keyring, nothing pasted by hand. It then guides installation. The wizard requires a running public ingress first.

PR tasks are tagged gh-pr:<owner>/<repo>#<n>, issue tasks gh-issue:…. A CHANGES_REQUESTED review or new head SHA reopens a done task; a merge marks it done. If flow was down, the wizard's Replay missed deliveries button replays the gap, deduped by delivery GUID.

Public ingress

Some connectors need an external HTTPS URL to reach flow inbound — Slack's OAuth redirect and GitHub's webhook POSTs. Public ingress is one reusable abstraction; flow keeps running locally, exposes only the connector callback paths (never the UI or data API), and still validates everything itself.

ProviderPublic URL fromUse when
none—Slack-only, accepting the localhost cert warning
zrokgenerated at runtimeyou want a public URL without running a server
manualFLOW_PUBLIC_BASE_URLyou already front flow with your own proxy/tunnel

zrok is recommended — flow embeds the zrok Go SDK (no binary, no subprocess). Enable once with zrok enable <token>, set FLOW_INGRESS_PROVIDER=zrok and a stable FLOW_ZROK_SHARE_NAME, and check live state at GET /api/ingress/status.

Your data — local, portable, yours

Everything lives under ~/.flow/ (override with $FLOW_ROOT). No server, no cloud, no telemetry — plain markdown beside a SQLite index, readable in any editor and versionable in git.

~/.flow/
  flow.db                          # SQLite index — projects, tasks, playbooks, owners
  kb/  user.md org.md products.md processes.md business.md
  projects/<slug>/  brief.md  updates/YYYY-MM-DD-*.md
  tasks/<slug>/     brief.md  updates/*.md  inbox.jsonl
  playbooks/<slug>/ brief.md  updates/*.md
  owners/<slug>/    charter.md updates/*.md

The database is an index, not the source of truth — every entity's real content is the markdown beside it; you could delete flow.db and rebuild from the markdown. Back up with git, Time Machine, or a synced folder (add kb/ to .gitignore if you push to a shared remote). To move machines: copy ~/.flow/ over, install the binary, run flow init once.

⚠️ Don't run flow on two machines simultaneously through a synced folder — SQLite doesn't tolerate concurrent writes from separate hosts.
